# 運用・セキュリティ簡易メモ（家計簿アプリ：PWA＋Supabase）

モバイルで**即入力**、PCで**閲覧・分析**。PWA（IndexedDB）＋Supabase同期。個人利用だが**複数ユーザー対応**の構え。

> 設定: 許可メール = **kikun_dev@gmail.com** / アプリ内PINロック = **ON**

---

## 1. データの範囲
- **対象**: 取引（transactions）、按分（transaction_splits）、カテゴリ（categories）、アカウント（accounts）、設定（user_settings）
- **非対象**: カード番号/口座番号等の**機微情報は保存しない**

## 2. アクセス制御・認証
- **認証**: Supabase Auth（**許可メールのみ**ログイン可）
  - 設定案A: **招待制**（Supabase Dashboardで新規サインアップ無効→Inviteで作成）
  - 設定案B: `allowed_emails` テーブルで許可リストを管理し、ログイン後に**不許可メールは即サインアウト**（簡易）
  - ひとまず許可するメール: **kikun_dev@gmail.com**
- **権限**: RLSで `user_id = auth.uid()` の行のみアクセス可
- **管理**: GitHub/Supabaseの**2段階認証**必須

## 3. 暗号化
- **通信**: HTTPS（Cloudflare Pages / Supabase）
- **保存**: Supabase側はマネージド暗号化。ローカルIndexedDBは**平文** → 代替として**アプリ内ロック（PIN）**でUIを保護（データ暗号化は将来）
- **秘密情報**: `service_role` は**サーバ側のみ**。クライアントは `anon key` と `url` のみ。Cloudflare Pagesの**環境変数**で管理

## 4. バックアップ/復元
- **正本**: Supabase（PostgreSQL）を正とする
- **バックアップ**: **月1回の手動エクスポート（SQL/CSV）**
- **復元手順**: 
  1) 誤削除発生時は直近エクスポートから対象テーブルを復元（範囲を限定）  
  2) 端末紛失時は**再ログイン**でサーバ正本を再同期  
  3) ローカル破損時はIndexedDBを初期化→サーバから再取得

## 5. ログ/監視
- **範囲**: 外部送信**なし**（デフォルト）。必要時のみSentryを**匿名化**で導入
- **監視**: 重大エラー時はUIトースト/再試行提案。Supabaseのダッシュボードでエラーログ参照
- **通知**: 重大障害時のみメール通知（任意）

## 6. 変更管理/リリース
- **ブランチ/タグ**: `main`/`dev` + SemVer（例：v0.1.0）
- **CI/CD**: Cloudflare PagesでPRプレビュー。マイグレーションは `/db/migrations` にSQL
- **ロールバック**: Cloudflare Pagesの直前ビルドに切替。DBは**後方互換**のスキーマ変更を基本

## 7. オフライン/同期運用
- **保存**: **ローカル即保存 → 同期キュー**（Background Sync）
- **競合**: **最新勝ち**（`updated_at` 比較）。編集中の不一致はダイアログで警告
- **キュー保持**: 7日（NFRに準拠）

## 8. データ保持/削除ポリシー
- **保持期間**: 無期限（個人利用）
- **削除**: 取引は**論理削除**（`is_deleted=true`）→後続で物理削除可能
- **端末紛失**: 管理画面から対象アカウントの**セッション失効**（再ログイン必須）

## 9. 依存/脆弱性管理
- **更新**: 月1回依存アップデート確認（Renovate任意）
- **監査**: GitHub Dependabot と `npm audit`
- **鍵**: 環境変数（Cloudflare/ Supabase側）。**漏えい時は即回転**

## 10. 端末/アプリ保護
- **アプリロック（PIN）**: **有効化（推奨）**  
  - **起動時**・**バックグラウンド復帰時**・**無操作60秒**でロック  
  - 実装：PINは**ハッシュ化**（PBKDF2 + ランダムソルト）して保存。UIガード（データ自体の暗号化は行わない/MVP）  
  - 連続誤入力ロックアウト（例：5回で30秒待機）
- **デバイス**: OSレベルの画面ロックを**必須運用**

## 11. インシデント対応
- **想定**: 誤削除/障害/不正アクセス
- **初動**: 影響範囲の切り分け→該当セッション停止→バックアップ確認
- **復旧**: エクスポートから復元/スキーマ修復。事後に原因と再発防止をメモ

## 12. エクスポート/インポート
- **MVP**: 未実装。将来はCSV/JSONを提供し、**パスワード付ZIP**で暗号化配布
- **レート**: 利用者数が増えた場合のみレート制御を検討

## 13. リスク登録簿（現時点）
| リスク | 影響 | 可能性 | 対応 |
|---|---|---|---|
| IndexedDB破損/互換 | 中 | 低 | 再ログインで再同期/バックアップ前提 |
| オフライン競合 | 中 | 中 | 最新勝ち+編集中ガード |
| サービスキー曝露 | 高 | 低 | クライアントに置かない/環境変数管理/回転手順 |
| 手動バックアップ忘れ | 中 | 中 | 月初リマインド（自分向け）/将来自動化 |
| 端末盗難 | 中 | 低 | OSロック必須/セッション失効 |
